The Big Picture

Before we get into the weeds, let's understand what Synaptum 8 actually is and how all its pieces talk to each other.

01 Three Tiers, One Job

Synaptum 8 has one job: help a neurologist turn a consultation into a formal letter. Everything in the system exists to support that workflow.

It's built as three separate layers that talk to each other:

Your Browser
Next.js :3001
The UI you click around in. Shows live progress.
HTTP / SSE
API Server
FastAPI :8891
Receives requests, validates, dispatches work.
SQLite DB (encrypted PHI)
Redis queue
Background Workers
Celery
Does the slow stuff: calls Claude, transcribes audio, OCRs PDFs.
A restaurant. The browser is the dining room where customers sit. The API server is the waiter who takes orders and brings food. The workers are the kitchen where the actual cooking happens. The waiter never cooks — they just pass orders to the kitchen and bring the result back.
Why not just have the API server do everything? Because calling Claude takes 10-15 seconds. If the API server waited around for that, it couldn't handle any other requests. Instead, it hands the work to a background worker and says "I'll let the browser know when it's done." That way the server stays fast and responsive.

02 The Consultation Workflow

The whole app is structured around a 3-step wizard. Each step maps to a phase of what the neurologist actually does:

Step 1
Patient
Who is this consultation for?
You do: Select or enter patient
System: Parses PHI automatically
Step 2
Sources
What happened in the consultation?
You do: Record audio, upload PDFs
System: Transcribes, OCRs, extracts PHI, redacts
Step 3
Letter
Generate the formal letter
You do: Answer questions, review, attest
System: Extracts, drafts, polishes letter

What "redacts everything" means

Before any text goes to an LLM (Claude), the system scrubs out personal information. Names become [NAME], Medicare numbers become [MEDICARE], addresses become [ADDRESS]. The only personal detail the LLM ever sees is the patient's first name (so it can write "I reviewed John today..." in the letter).

This is critical to understand: The LLM never sees raw patient data. It works with redacted text only. This is the PHI containment model — we'll go deep on it in Module 5.

03 How the Browser Knows What's Happening

Here's a question: if the API server hands work to a background worker and immediately returns, how does the browser know when the letter is ready?

The answer is SSE (Server-Sent Events). Think of it like a one-way radio channel:

Browser
1. POST "Start generating a letter for session X"
API Server
Browser
2. Response "Here's your task ID: abc123"
Server
server dispatches to celery worker
Browser
3. SSE Stream "I'll watch task abc123 for updates"
Worker
worker publishes progress via redis
10%organizingReading and structuring the sources
40%generatingDrafting the letter
70%refiningPolishing tone and grammar
100%completeDone! Letter ID: xyz — browser fetches the result
Ordering pizza online. You place the order (step 1), get a confirmation number (step 2), then the pizza tracker shows you live updates — "preparing", "in the oven", "out for delivery" (steps 3-4). You don't keep calling the store to ask "is it ready yet?"

04 The Letter Pipeline

This is the heart of the system. When you click "Generate Letter", four separate LLM calls happen in sequence:

Stage 1
Extract
Reads transcript + docs, organizes into structured sections
Stage 2
Draft
Writes the letter from clean, structured data
Stage 3
Polish
Refines tone, grammar, Australian English
Stage 4
Actions
Extracts follow-up tasks (optional)

Each stage is a separate LLM call — 4 calls total. Breaking it up gives better results because each call has a focused job.

Why not do it all in one LLM call? Breaking it into stages gives better results. Each stage has a focused job with its own prompt. It also means you can cache stage 1's output (the extraction) — which is exactly what the interview system does. The extraction runs once during the interview, then gets reused when the letter is generated.

05 The File Layout (Where Things Live)

api/ — Backend (Python)
main.py — starts the server
core/ — config, security, middleware
db/ — models and connections
endpoints/ — the "waiter" layer
services/ — the "kitchen" layer
tasks/ — the "slow cooking" layer
web/ — Frontend (TypeScript)
app/ — pages and layouts
components/ — UI components
lib/api/ — talks to the backend
lib/stores/ — app state
lib/hooks/ — reusable logic (SSE, etc.)

Don't worry about memorising this. The key insight is the three layers in the backend: endpoints (receive requests) → services (do the work) → models (touch the database). We'll go deep on this in Module 2.

06 The External Services

Your system doesn't do everything itself. It calls five external services:

ServiceWhat it doesWhere it runs
Claude APIGenerates letters, extracts clinical info, creates interview questionsAnthropic's cloud (or AWS Bedrock)
MedASR+Medical speech recognition — turns consultation audio into textLocal Docker container, port 8002
AssemblyAIAlternative transcription service (cloud-based fallback)Cloud API
PaddleOCRReads text from uploaded PDFs and scanned documentsLocal Docker container, port 8010
OllamaLocal LLM fallback (runs smaller models when Claude is unavailable)Local Docker container, port 11434

Each of these can fail or go down. That's why every external call goes through a circuit breaker — a safety mechanism that stops the system from hammering a dead service. We'll cover circuit breakers in Module 5.
