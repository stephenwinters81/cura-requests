The Frontend

Next.js pages, the imaging request form, admin dashboards, and how the UI talks to the server.

01 What is Next.js?

Before we look at the pages, let's understand the tool that builds them.

React is a JavaScript library for building user interfaces. It lets you break a page into small, reusable pieces called components — a button, a dropdown, a form field. You snap these together to build a full page. React was created by Facebook and is the most popular way to build web apps today.

Next.js is a framework built on top of React. A framework is like a pre-built house frame — it gives you structure, plumbing, and wiring so you can focus on the rooms and furniture. Next.js adds things React doesn't have on its own: page routing, server-side rendering, and server actions (we'll explain those shortly).

App Router: folders become URLs

Requests uses Next.js 14 with the App Router. This means every page in the app is defined by its folder structure in the codebase. The file path maps directly to the URL the user visits in their browser.

For example:

File pathURL
app/login/page.tsx/login
app/requests/new/page.tsx/requests/new
app/admin/practices/page.tsx/admin/practices

The file must be called page.tsx. The folder it sits in determines the URL. Simple as that.

Server-side rendering

Server-side rendering (SSR) means the server prepares the full page before sending it to your browser. The browser receives a complete page that's ready to display, rather than an empty shell that has to build itself using JavaScript.

This matters for two reasons: speed (the page appears faster because it arrives ready-made) and security (sensitive logic runs on the server, not in the browser where anyone could inspect it).

A restaurant menu. Each page of the menu is a route (a URL). The kitchen (server) prepares the dish (page) before bringing it to your table (browser). You don't have to cook it yourself — it arrives ready to eat.

03 The Imaging Request Form

This is the heart of the entire application. The page at /requests/new is where staff spend most of their time. Everything else in the system exists to support what happens when this form is submitted.

Here are all the fields, in order:

1 Radiology Practice A searchable dropdown (called a combobox). Start typing and matching practices appear. Recently-used practices are sorted to the top — sorted by how often each one has been used (usageCount). If the practice isn't in the list, you can type a name manually.
2 Exam Type A dropdown with 27 predefined exam types (like "MRI Brain", "CT Chest", "Ultrasound Abdomen") plus an "Other" option. Selecting "Other" reveals a free-text field where you can type the exam name.
3 Patient Details A large text area where staff paste the patient's information — name, date of birth, Medicare number, phone, address. This is the PHI dump that gets parsed server-side by phi-parser.ts (covered in Module 3).
4 Clinical Details Free text explaining why the imaging is needed. For example: "Persistent headaches, rule out space-occupying lesion." This goes on the PDF that the radiology practice receives.
5 Contrast Reaction A Yes/No toggle. Some imaging scans use contrast dye. If the patient has had a reaction to contrast dye before, this must be flagged.
6 eGFR An optional number field. eGFR is a kidney function measurement. The radiology practice needs this when contrast dye is involved, because poor kidney function can make contrast dye dangerous.
7 Provider A dropdown showing the referring doctor + their location. There are 7 combinations across 3 doctors. The logged-in user's default provider is pre-selected.
8 Report by Bhatia A checkbox. When ticked, the request specifies that the imaging report should be done by a particular radiologist (Dr Bhatia).
9 Patient Email Optional. If provided, the patient receives a copy of the request by email.
10 Delivery Method How to send the request to the radiology practice: email, fax, or both. The practice's preferred method is pre-selected from their record in the database.

Smart features that save time

Auto-save to localStorage. As you fill out the form, every field is continuously saved to localStorage — a small storage area built into your browser. If the browser crashes, your internet drops, or your session times out, the form data survives. When you come back, the form is restored exactly where you left off. No lost work.

Keyboard navigation. Staff submit around 30 requests per week. The form is optimised for speed — you can tab between fields, type to search dropdowns, and submit without touching the mouse.

Duplicate detection. If you submit a request for the same patient and same exam type within 24 hours, the app shows a warning: "You already submitted this combination earlier today. Continue anyway?" This catches accidental double-submissions.

A smart order form at a cafe. It remembers your usual order (auto-save), suggests your favourite drink first (recently-used practices), and warns you if you accidentally order the same thing twice (duplicate detection).

04 Server Actions vs API Routes

There are two ways to get data into the Requests system. Both end up doing the same thing, but they're designed for different users.

Web UI
Server Actions
Staff fill out the form in their browser. When they click Submit, the form data goes directly to a server function — no separate API call needed.

Synaptum 8 sends a JSON request to /api/v1/requests. Authenticated with an API key.

Server Actions are a Next.js feature. When the staff member clicks "Submit" on the form, the browser sends the data directly to a function that runs on the server. There's no separate API endpoint involved — the form and the server function are connected directly by the framework. This is simpler and more secure because the server function is never exposed as a public URL.

API Routes are traditional REST endpoints under /api/v1/. These exist for Synaptum 8 — the letter-generation system — which needs to create imaging requests programmatically (without a human clicking buttons). API routes use API key authentication, rate limiting, and scoped permissions (covered in Module 6).

Both paths do the same thing: validate the data, save it to the database, generate a PDF, and queue a delivery job. The only difference is the front door.

Two doors to the same kitchen. The dining room door (server actions) is for walk-in customers — staff using the web form. The delivery entrance (API routes) is for Uber Eats — Synaptum 8 placing orders programmatically. Same chef, same food, different entrance.

05 UI Components

The interface is built from two tools working together: Radix UI and Tailwind CSS.

Radix UI provides accessible, unstyled primitives. "Primitives" means basic building blocks — a dropdown, a dialog box, a toggle switch, a combobox. "Unstyled" means they have no visual design. They handle the hard parts like keyboard navigation, screen reader support, and focus management, but they look like nothing until you style them. "Accessible" means they work for people using assistive technology (screen readers, keyboard-only navigation).

Tailwind CSS provides the visual styling. Instead of writing traditional CSS files, you add small utility classes directly to HTML elements. For example, bg-white p-4 rounded-lg shadow means "white background, padding of 4 units, rounded corners, drop shadow." It's like painting by numbers — each class does one small thing.

LEGO bricks. Radix UI provides the shapes — a button brick, a dropdown brick, a dialog brick. Tailwind CSS is the paint. You snap the bricks together and paint them to build the interface.

06 How the UI Knows Delivery Status

After a staff member submits the form, what happens next? The form doesn't send the email or fax directly — it queues a background job (covered in Module 5). So the user needs a way to check on the status of their request.

Immediately after submission

The user sees a success page with the request ID and a confirmation that the delivery has been queued. This happens almost instantly — the app doesn't wait for the email or fax to actually be sent.

The delivery timeline

Each request has a detail page at /requests/[id]. This page shows a delivery timeline — a visual progression of what happened to each delivery job:

Status 1
Queued
Job created, waiting in BullMQ
Status 2
Processing
Worker has picked up the job
Status 3
Sent
Email/fax dispatched, awaiting confirmation
Status 4
Delivered
Confirmed received (or Failed if not)

If a delivery fails, the status shows as red with an error message. The worker automatically retries with exponential backoff (up to 3 attempts), but if all retries fail, the delivery appears as permanently failed.

The dashboard: failed deliveries at a glance

Staff don't have to check each request individually. The dashboard (/dashboard) shows two things:

Recent submissions — the last few requests, so you can quickly spot-check that things went through.

Failed deliveries — any requests where delivery failed after all retry attempts. These are flagged for immediate attention. A staff member can view the request detail, see what went wrong, and manually trigger a re-send if needed.

The page refreshes to check status. The delivery timeline is not a live real-time feed. Staff refresh the page (or navigate back to it from the history table) to see the latest status. For a system handling ~30 requests per week, this is more than sufficient — no need for WebSockets or push notifications.

The history table at /requests shows all past requests with search and filters. You can search by patient name, filter by date range, filter by delivery status (delivered, failed, pending), or filter by radiology practice. Clicking any row takes you to the detail page.
