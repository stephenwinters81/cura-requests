The Big Picture

What Requests does, who uses it, and how all the pieces fit together.

01 What Requests Does

CURA Medical Specialists is a small medical practice in Australia with three referring doctors. When a doctor wants a patient to get an imaging scan — like an MRI of the brain or a CT scan of the chest — they need to send a formal imaging request to a radiology practice.

That request is basically a form that says: "Here is the patient, here is what we need scanned, and here is the doctor ordering it." The radiology practice receives it, books the patient in, and performs the scan.

The Old Way

Previously, CURA used two separate services stitched together:

Snapforms — an online form builder. Staff filled out a web form with the patient's details and what scan was needed. Think of it like Google Forms, but for medical paperwork.

Make.com — an automation service. When the form was submitted, Make.com would automatically generate a PDF document and send it to the radiology practice via email or fax. Think of it like a robot assistant that watches for new form submissions and carries out a checklist of steps.

This worked, but it had real problems:

ProblemWhat it meant
No control over PDF layoutThe generated PDF looked however Snapforms decided. You couldn't customise it to look professional or match CURA's branding.
Credits per executionMake.com charges money every time the automation runs. With ~30 requests per week, costs added up.
Fragile fax lookupMake.com used OpenAI (an AI service) to look up fax numbers. This sometimes broke, causing outages where no requests could be sent.
No visibilityIf something failed, there was no easy way to see what went wrong or retry it.

The New Way

Requests replaces both Snapforms and Make.com with a single self-hosted application. "Self-hosted" means it runs on a server that CURA controls, rather than relying on someone else's service. This gives CURA full control over:

  • The form that staff fill out
  • The PDF that gets generated
  • How and when delivery happens (email or fax)
  • The list of radiology practices and their contact details

No more per-execution fees. No more fragile AI lookups. No more mystery failures.

02 Who Uses It

Requests has a very small user base. This isn't a public website — it's an internal tool for one medical practice.

The Humans

Three referring doctors: Winters, Harrison, and Khalil. Each doctor has provider numbers — unique identifiers issued by Medicare that are tied to specific practice locations. One doctor might have different provider numbers for different clinics they work at. Across the three doctors, there are 7 provider number/location combinations in total.

A provider number is like a work ID badge, but location-specific. If Dr Winters works at both the city clinic and the suburb clinic, she has a separate badge (provider number) for each one. The imaging request needs to include the right badge for where the patient was seen.

Admin staff at CURA also use the system. They typically fill out the form on behalf of the doctors, manage the list of radiology practices, manage user accounts, and handle any delivery issues.

The Machine

Synaptum 8 is CURA's practice management system — the software they use day-to-day to manage patient records, appointments, and billing. Synaptum 8 can also create imaging requests automatically via the API.

An API (Application Programming Interface) is a way for one piece of software to talk to another. Instead of a human filling out a form in a browser, Synaptum 8 sends a structured message to Requests saying "create this imaging request with these details." The result is the same — a PDF gets generated and delivered — but no human had to fill out a form.

The key thing to notice: the user doesn't wait for the email or fax to send. The moment the server saves the request and puts it on the queue, the user sees a success message. Delivery happens in the background.

A pizza shop. The front counter (browser) takes your order. The kitchen (server) prepares it and puts a ticket on the rail (queue). The delivery driver (worker) picks it up and takes it to your door (email/fax). The driver doesn't hold up the counter — they work independently. If the driver can't find your house the first time, they'll try again.

04 The Tech Stack

A tech stack is the collection of technologies used to build an application. Each piece handles a specific job. Here's what Requests is built with:

TechnologyWhat it does
Next.js 14 (App Router) The web framework. It handles the pages you see in your browser, the forms you fill out, and the API routes that Synaptum 8 talks to. Think of it as the foundation that everything else is built on.
PostgreSQL + pgcrypto The database. PostgreSQL stores all the data (requests, patients, practices). pgcrypto is an add-on that encrypts sensitive fields — so even if someone got access to the database, patient details would be unreadable scrambled text.
Prisma An ORM (Object-Relational Mapper). Instead of writing raw database queries, developers write TypeScript code and Prisma translates it into database language. Like a translator between the app and the database.
Auth.js v5 + TOTP Handles login and security. TOTP stands for Time-based One-Time Password — those 6-digit codes from an authenticator app. Every user must set up 2FA (two-factor authentication). No exceptions.
BullMQ + Redis The job queue system. BullMQ manages the list of delivery jobs (send this email, send that fax). Redis is the fast in-memory database that BullMQ uses behind the scenes to track the queue.
React-PDF Generates PDF documents. When a request is submitted, React-PDF creates a professional-looking imaging request form as a PDF file.
Nodemailer Sends emails. It connects to Google Workspace's email servers (SMTP) to deliver imaging requests by email.
Notifyre An Australian fax API service. Many radiology practices still use fax (yes, really). Notifyre is ISO 27001 certified and healthcare-compliant — meaning it meets strict security standards.
BinaryLane VPS The server itself. A VPS (Virtual Private Server) hosted in Sydney, Australia. All data stays on this server, in Australia.

05 Key Design Decisions

Three design decisions shape everything about how Requests works. Understanding these will help every other module make sense.

Insight: Everything is async.
"Async" is short for asynchronous — it means "not happening at the same time." When you submit a request, the delivery (email or fax) does not happen right then and there. Instead, a delivery job is placed on a queue, and a separate background process (the worker) handles it later — usually within seconds, but independently of your browser session. This means the app stays fast and responsive. If a fax takes 30 seconds to send, you don't sit there staring at a loading spinner. You get your confirmation instantly, and the worker handles the rest.
Remember: All data stays in Australia.
The server is in Sydney. The fax provider (Notifyre) is Australian. Email goes through Google Workspace SMTP. At no point does patient data leave the country. This isn't a nice-to-have — it's a legal requirement under the Australian Privacy Act. The practice handles PHI (Protected Health Information), and Australian law governs how and where that data can be stored and transmitted.
Warning: PHI is everywhere.
PHI — Protected Health Information — includes patient names, dates of birth, Medicare numbers, phone numbers, addresses, and anything else that could identify a patient. In this app, PHI is encrypted at rest (scrambled in the database so it can't be read without the encryption key), never written to log files, and carefully controlled in every part of the system. Mishandling PHI isn't just a bug — it's a potential breach of the Privacy Act, which can result in serious legal consequences. Every design decision in this app considers PHI.
