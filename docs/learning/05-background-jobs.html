Module 05 / 07

Background Jobs & Delivery

How Requests sends emails and faxes without making users wait. The queue, the worker, retries, and webhook confirmations.

Section 1

Why Background Jobs?

Here is the problem. Sending a fax can take 30 seconds or more. An email server might be temporarily down. If the web server tried to do all of that during the page request, the user would stare at a loading spinner for minutes.

That is a terrible experience. And it gets worse: what if three deliveries need to happen for one request? The user would wait for all of them, one after another.

The solution is a job queue. When someone submits an imaging request, the web server does not send anything itself. Instead, it creates "jobs" — small packets of work — and drops them onto a queue. A completely separate process picks up those jobs and handles the actual sending.

The user sees a confirmation instantly. The deliveries happen in the background.

Analogy

Think of a restaurant kitchen with a ticket rail. The waiter (web server) takes the customer's order and clips the ticket to the rail. Then the waiter immediately goes back to serve more customers. The cook (worker) picks up tickets one by one and prepares the food. If a dish burns, the cook remakes it. The waiter does not need to stand in the kitchen and watch.

Redis is an in-memory data store. Think of it as a super-fast message board. It lives entirely in your server's RAM, which makes it extremely quick to read and write. BullMQ uses Redis to store the job queue. When the Next.js server adds a job, it writes to Redis. When the worker checks for new jobs, it reads from Redis.

BullMQ is a Node.js library for managing job queues. It handles: adding jobs to the queue, distributing them to workers, tracking each job's status, and retrying jobs that fail. It takes care of all the hard coordination so our code just says "add this job" and "process jobs like this."

PM2 is a process manager. It keeps the delivery worker running as its own process, completely separate from the Next.js web server. If the worker crashes, PM2 restarts it automatically. This separation means a worker crash never takes down the website, and a website restart never interrupts an in-progress delivery.

Key point: The web server and the worker are two separate processes. They communicate through Redis, not directly. This is what makes the system resilient.

Section 3

Job Types

Each delivery is a separate job with its own type. Here are the four types of jobs the system creates:

Type What it does Destination
provider_email Emails the PDF to the radiology practice Practice email address
provider_fax Faxes the PDF via the Notifyre API Practice fax number
filing_email Emails a PDF copy to CURA for their records clinic@curaspecialists.com.au
patient_email Emails the PDF to the patient (optional) Patient's email address

A single imaging request might create 2, 3, or 4 jobs depending on the delivery method and whether a patient copy is requested. At minimum, there is always a provider delivery and a filing copy. If the practice wants both email and fax, that is three jobs. Add a patient copy and you are at four.

Section 4

The Worker's Workflow

When the worker picks up a job, it follows this pipeline step by step:

  1. Pick up the next job from the Redis queue
  2. Set the DeliveryJob status to "processing" in the database
  3. Load the encrypted PDF from disk and decrypt it into memory
  4. Send via the correct channel — Nodemailer for email, Notifyre API for fax
  5. On success: update status to "sent" (email) or wait for webhook confirmation (fax)
  6. On failure: increment the attempt counter and log the error
  7. Update the parent ImagingRequest's overall status based on all its delivery jobs
  8. If a webhook is configured: notify Synaptum 8 about the status change

The worker processes one job at a time to keep things simple and predictable. If it fails on step 4, it does not skip to step 7. It jumps to step 6 (failure handling) and the job goes back in the queue for a retry.

Section 5

Retries & Exponential Backoff

Things fail. An email server goes down. A fax line is busy. The internet hiccups. The system is designed to handle this gracefully.

Each job gets a maximum of 3 attempts. After each failure, the system waits before trying again — but the wait gets longer each time. This is called exponential backoff.

30 sec
Retry 1
2 min
Retry 2
10 min
Retry 3

Why exponential? If an email server is down, hammering it every second makes things worse. It can overload a server that is already struggling to recover. Spacing out retries gives the remote system time to come back online.

Analogy

Calling a friend who is not answering. You try again after 30 seconds. Still no answer? Wait 2 minutes. Still nothing? Wait 10 minutes. After 3 tries, you leave a voicemail (mark the job as "failed") and deal with it manually.

After all 3 attempts fail, the job stays in the database with a status of "failed". An admin can see it on the dashboard and manually re-send it.

Section 6

Fax Confirmations (Webhooks)

Email and fax are confirmed differently.

Email is "fire and forget." Once the system hands the email to the SMTP server, it is marked as "sent". We trust the email infrastructure to handle the rest. There is no callback.

Fax is different. When the worker sends a fax via the Notifyre API, the fax does not arrive instantly. It has to dial, negotiate with the receiving machine, and transmit page by page. This can take minutes. So the worker marks the job as "sent" and moves on.

Later, when the fax actually arrives (or fails), Notifyre calls back to a special endpoint on our server:

/api/fax/webhook

This is called a webhook. It is Notifyre saying "Hey, that fax you asked me to send? It was delivered." (Or "It failed.") Our server then updates the DeliveryJob status to "delivered" and records a confirmedAt timestamp.

But how do we know the webhook actually came from Notifyre and not an attacker? HMAC verification. Notifyre signs every webhook with a secret key. Our server checks the signature before trusting the message. If the signature does not match, the webhook is rejected.

Analogy

Regular email is like dropping a letter in a mailbox. You trust it will arrive. Fax is like registered mail. The post office calls you back to confirm delivery. And the HMAC signature is like the post office using a secret codeword so you know the call is real.

Section 7

Idempotency

There is a word that comes up a lot in job queue systems: idempotency. It means "doing the same thing twice produces the same result as doing it once."

Why does this matter? Imagine the worker picks up a job, sends the fax, but crashes before it can mark the job as complete. When PM2 restarts the worker, BullMQ might think the job was never finished and try to run it again. Without protection, the patient would receive two identical faxes.

BullMQ prevents this by giving each job a unique ID. The worker tracks which jobs it has processed. If a job ID has already been completed, it will not be processed again — even if the worker restarted.

This is a safety net. It means the system can be aggressive about retries without worrying about duplicate deliveries.

In short: Crashes and restarts are normal in production. Idempotency ensures they never cause duplicate faxes or emails. Each job runs exactly once, even in the worst case.
