Module 06 of 07

The API Layer

How Synaptum 8 talks to Requests without a human touching the web form. API keys, scopes, rate limits, and webhooks.

Section 1

What is the API For?

CURA's practice management system is called Synaptum 8. It is a FastAPI Python application that handles appointments, billing, clinical notes, and more.

Synaptum 8 needs to create imaging requests. But it is a machine. It does not have a human sitting in front of the web form, clicking buttons and typing patient details. It needs a way to talk to Requests programmatically — meaning through code, not through a browser.

That is what the API is for. API stands for Application Programming Interface. It is a set of rules that lets one piece of software talk to another.

The Requests web form is the "front door" for humans. The API is a back door for machines.

Analogy

A drive-through window. The main entrance (web form) is for humans. The drive-through (API) is for another system (Synaptum 8) to place orders without coming inside. Same kitchen, same food — just a different window.

The API lives at a specific URL, called the base URL:

https://requests.curaspecialists.com.au/api/v1

Every API endpoint starts with this base URL. The /v1 part means "version 1." If we ever need to make breaking changes, we can create /v2 without disrupting Synaptum 8.

"REST" is the style of API we use. It means each URL represents a "resource" (like a request or a practice), and you interact with it using standard HTTP methods like GET (read) and POST (create).

Section 2

Authentication — API Keys

When a human logs into Requests, they use a username and password (plus MFA). But Synaptum 8 is not a human. It does not have fingers or a phone for MFA codes.

Instead, it uses an API key. This is a long, random string that acts like a password for machines.

API keys in Requests have a specific format:

  • Production: rq_live_ followed by random characters
  • Development: rq_test_ followed by random characters

The prefix makes it obvious at a glance whether a key is a real production key or a test key. You would never accidentally use a test key in production (or vice versa).

How does Synaptum 8 send the key? In an HTTP header with every request:

Authorization: Bearer rq_live_abc123def456...

The word "Bearer" is a standard convention that means "I am the bearer (holder) of this token — let me in."

How are keys managed?

  • A key is shown once at creation. After that, only the prefix is visible (like rq_live_abc1...). There is no way to see the full key again.
  • In the database, the key is stored as a SHA-256 hash. This is a one-way scramble — just like a password. If someone breaks into the database, they still cannot recover the key.
  • Each key is linked to a user account. This creates an audit trail — we always know which user created which key and which requests were made with it.
  • Keys can be expired (set a date when they stop working), revoked (turned off immediately), and scoped (limited to certain actions).
Analogy

A hotel key card. You get it at check-in (creation). It only opens your room (scoped access). It expires at checkout (expiry). And the front desk can deactivate it anytime (revoke).

Section 3

Scopes — What Can a Key Do?

Not every API key should be allowed to do everything. Scopes control exactly what actions a key is permitted to perform.

Think of scopes as a checklist of permissions. When you create an API key, you tick which permissions it gets.

Scope Allows
requests:write Create imaging requests, re-send failed deliveries
requests:read View request status and delivery details
practices:read List and search radiology practices
providers:read List providers and exam types
practices:write Create/update practices (optional, rare)

A key with only requests:read can check the status of a request. But it cannot create a new one — it would need requests:write for that.

This follows the principle of least privilege: give every key only the permissions it actually needs. If Synaptum 8 only needs to create requests and check their status, it gets requests:write and requests:read — nothing more.

Section 4

The Endpoints

An endpoint is a specific URL that does a specific thing. Each one has:

  • A method (GET = read, POST = create)
  • A path (the URL after the base URL)
  • A required scope (what permission the key needs)

Here is every endpoint in the Requests API:

Method Path Scope What it does
POST /requests requests:write Create imaging request (validates, parses PHI, generates PDF, queues delivery)
GET /requests requests:read List requests (paginated, filterable)
GET /requests/:id requests:read Get request status + delivery job statuses
POST /requests/:id/resend requests:write Re-queue failed delivery jobs
GET /requests/:id/pdf requests:read Download the generated PDF
GET /practices practices:read List radiology practices
GET /providers providers:read List providers
GET /exam-types providers:read List exam types

The :id in paths like /requests/:id is a placeholder. In a real request, you would replace it with the actual request ID, like /requests/abc-123-def.

Response format. Every response from the API follows the same shape:

// Success { "ok": true, "data": { ... } } // Error { "ok": false, "error": { "code": "...", "message": "..." } }

This makes it easy for Synaptum 8 to check: if ok is true, the request worked. If false, something went wrong, and the error tells you what.

Key Rule

PHI is accepted via the API (in the rawPhiInput field) but never returned in API responses. You can send patient data in, but you can only get back metadata and delivery status. This is a deliberate security design — the API is not a way to extract patient data.

Section 5

Rate Limiting

Rate limiting prevents any single API key from overwhelming the system with too many requests at once. There are two limits:

  • General limit: 120 requests per minute per API key (for all endpoints)
  • Create limit: 30 requests per minute for creating new imaging requests (the POST /requests endpoint)

The create limit is lower because creating a request does a lot of work: validation, PHI parsing, PDF generation, and queuing delivery jobs. It is much heavier than a simple read.

Every API response includes three special headers that tell the caller where they stand:

Header Meaning
X-RateLimit-Limit Maximum requests allowed per window
X-RateLimit-Remaining How many requests are left in the current window
X-RateLimit-Reset When the window resets (Unix timestamp)

If you exceed the limit, the API responds with HTTP status code 429 (Too Many Requests). Synaptum 8 should wait and try again after the reset time.

Analogy

A speed limit. You can drive (make requests) as fast as you want up to the limit. Go over, and you get a ticket (HTTP 429 error) and have to wait.

Section 6

Webhooks — Pushing Status Back

So far, the communication is one-way: Synaptum 8 sends requests to the API. But what happens after that? Synaptum 8 needs to know when a fax has been delivered, or when an email bounced.

There are two ways to find out:

  1. Polling: Synaptum 8 asks "is it done yet?" every few minutes. This wastes resources and adds delay.
  2. Webhooks: Requests sends a notification to Synaptum 8 the moment something changes. This is instant and efficient.

We use webhooks. Here is how they work:

  • When you create an API key, you can configure a webhook URL — a URL on Synaptum 8's side that accepts incoming notifications.
  • When a delivery status changes, Requests sends an HTTP POST to that URL with the event details.
  • The three events that trigger webhooks are:
Event When it fires
delivery.sent Email or fax was successfully sent to the provider
delivery.delivered Delivery confirmed (e.g., fax received by the radiology practice)
delivery.failed Delivery failed after all retries were exhausted

How does Synaptum 8 know the webhook really came from Requests? Every webhook is signed with HMAC-SHA256. This is a cryptographic signature that proves the notification was generated by Requests and has not been tampered with. Synaptum 8 verifies this signature before trusting the payload.

Analogy

A tracking notification. Instead of Synaptum 8 checking "is my package delivered yet?" every 5 minutes (polling), Requests sends a text message (webhook) the moment something changes. The text comes from a verified number (HMAC signature) so you know it is real.

Section 7

Duplicate Detection

Mistakes happen. A network glitch might cause Synaptum 8 to send the same request twice. A bug could trigger a duplicate submission. Without protection, the same patient could get the same imaging request sent to the same radiology practice twice.

To prevent this, the API has automatic duplicate detection.

If Synaptum 8 sends a request with the same patient and same exam type within a 24-hour window, the API does not create a duplicate. Instead, it responds with:

// HTTP 409 Conflict { "ok": false, "error": { "code": "DUPLICATE_REQUEST", "message": "A matching request was created within the last 24 hours" } }

HTTP 409 means "Conflict" — the new request conflicts with an existing one.

But sometimes a duplicate is intentional. Maybe the first request had wrong information and needs to be re-done. In that case, Synaptum 8 can include "force": true in the request body to override the duplicate check.

How it works

The API checks for matching patient details and exam type within the last 24 hours. If a match is found, it returns 409 Conflict. Include "force": true in the request body to override this check and create the request anyway.
