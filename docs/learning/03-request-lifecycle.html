Module 03 · ~12 min

The Request Lifecycle

Follow a single imaging request from the moment a staff member clicks Submit all the way through to the outgoing delivery queue. Seven stages, one continuous pipeline.

Section 1

The Journey of a Request

Every imaging request goes through seven stages in a fixed order. Each stage does one job, then hands off to the next. If any stage fails, the system stops and tells you what went wrong. Nothing slips through silently.

Here is the full pipeline, from button click to outgoing mail:

1

Form Submit

A staff member fills in the practice, exam type, referring doctor, and patient details, then clicks Submit. The browser sends all that data to the server.

2

Validation

The server checks every single field using a library called Zod. Is the exam type one of the 28 allowed values? Is the patient text present? Is the radiology practice valid? If anything is wrong, the server rejects the submission instantly.

3

Duplicate Check

Has the same patient already been submitted for the same exam type within the last 24 hours? If so, the system shows a warning. This prevents accidental double-ups.

4

PHI Parsing

The patient details text blob is fed through a parser that uses regex patterns to extract structured fields: name, date of birth, Medicare number, phone, address, email. More on this in Section 2.

5

Save & Encrypt

The request is saved to the database. All patient information fields are encrypted using pgcrypto before they touch the disk. Even if someone accessed the database directly, the PHI would be unreadable.

6

PDF Generation

A professional A4 imaging request form is rendered as a PDF, complete with the CURA logo, patient details, exam details, and referring doctor signature. The PDF is encrypted and saved to disk. More on this in Section 3.

7

Queue Delivery

One or more delivery jobs are created in BullMQ for each delivery channel: email, fax, filing copy, and optionally a patient copy. The HTTP response returns immediately. Deliveries happen in the background. More on this in Section 4.

Analogy: The Post Office

Think of this like a post office. You write a letter (the form). The clerk checks the address is valid (validation). They make sure you haven't already sent the same letter today (duplicate check). They sort the contents into neat categories (PHI parsing). They seal it in an envelope (encryption). They print a professional label (PDF generation). And finally, they put it in the right outgoing bins -- regular mail, express, a copy for your files (queue delivery).

Section 2

PHI Parsing -- The Smart Clipboard

PHI stands for Protected Health Information. When staff create a request, they paste (or type) the patient's details as a single block of text. It might come from Synaptum, from a referral letter, or from the patient's file. It looks messy. That is fine.

A file called phi-parser.ts on the server reads that text and uses pattern matching to pull out structured fields. It works like a smart clipboard that knows what it is looking at.

Here is what it extracts:

  • NAME First line with 2 or more words
  • DOB Dates in DD/MM/YYYY, DD-MM-YYYY, and similar formats
  • MEDICARE 10-digit Medicare numbers (with or without spaces)
  • PHONE Mobile (04xx) and landline patterns
  • EMAIL Standard email addresses
  • ADDRESS Addresses with state abbreviation and 4-digit postcode

Here is an example of what goes in and what comes out:

Raw text input (what the staff member pastes) 
John Smith
DOB: 15/03/1965
Medicare: 2345 67890 1
0412 345 678
john.smith@email.com
42 Wallaby Way, Sydney NSW 2000
↓ phi-parser.ts ↓
Parsed output (structured data) 
name:      "John Smith"
dob:       "1965-03-15"
medicare:  "2345678901"
phone:     "0412345678"
email:     "john.smith@email.com"
address:   "42 Wallaby Way, Sydney NSW 2000"

Two important things to know:

Both versions are saved. The original raw text is stored in a field called rawPhiInput and the parsed result goes into parsedPhi. Both fields are encrypted in the database.

If parsing fails, nothing is lost. Say the text is weirdly formatted and the parser cannot find a Medicare number. That is okay. The raw text is always preserved as a fallback. The system does not throw an error or reject the submission. It gracefully continues with whatever it managed to extract.

Why not just use structured form fields?

Because staff are copying and pasting from other systems (like Synaptum 7) where the data comes as a text block. Forcing them to split it into separate fields would slow them down. The parser does that work automatically.

Section 3

PDF Generation

Once the data is saved and encrypted, the system generates a professional PDF. This is the actual document that gets emailed or faxed to the radiology practice.

The PDF is built using a library called @react-pdf/renderer. Instead of manually drawing lines and text on a page, developers write React components that describe the layout, and the library turns those into a PDF file. It is like a template that gets filled in with each request's data.

The A4 document has these sections:

  • Header CURA Medical Specialists logo and practice details
  • To block The radiology practice name and address
  • Patient details Name, DOB, Medicare, phone, address (from the parsed PHI)
  • Exam request The imaging exam type and any clinical notes
  • Referring doctor Doctor name, provider number, and signature
  • Footer Request ID, date, and contact information

The PDF is generated entirely on the server. It never passes through the user's browser. Once generated, the PDF is:

1. Encrypted with AES-256 -- a strong encryption standard -- so the file on disk is unreadable without the decryption key.

2. Saved to disk on the server, where it waits to be attached to delivery jobs.

Analogy: The Printing Press

Think of it as a printing press. You feed in the data (patient details, exam type, doctor). The press stamps out a professional-looking form. Then the form gets locked in a safe (encrypted on disk) until someone with the right key needs to view or send it.

Section 4

The Handoff to the Queue

With the data saved and the PDF generated, the system is ready to deliver. But it does not send anything right now. Instead, it creates jobs in a queue.

A queue is like a to-do list. Each delivery channel gets its own job. A separate background process called the worker picks up these jobs one by one and handles the actual sending.

There are four possible delivery channels:

  • EM

    provider_email

    The PDF is emailed to the radiology practice. Uses Nodemailer via Google Workspace SMTP.

  • FX

    provider_fax

    The PDF is faxed to the radiology practice via Notifyre, an Australian fax API. Costs about 3 cents per page.

  • FL

    filing_email

    A copy is emailed to clinic@curaspecialists.com.au for the clinic's records.

  • PT

    patient_email

    An optional copy is emailed to the patient. Only created if the patient's email address was provided.

Key Insight

The HTTP response returns immediately with a "success" message. The staff member does not wait for emails to send or faxes to connect. Deliveries happen later, in the background. This is called asynchronous processing, and it is the reason the form feels fast even though there is a lot happening behind the scenes.

If a delivery fails (say the fax line is busy), the worker will retry automatically. The staff member does not need to do anything. They can check the request's delivery status later to see if everything went through.

Section 5

What the User Sees

From the staff member's perspective, the whole process feels simple and fast:

1. They fill in the form -- pick a practice, choose the exam type, paste patient details, select the referring doctor.

2. They click Submit.

3. They see a success page with the request ID and a "View/Print PDF" link. This appears within a couple of seconds.

Behind the scenes, all seven pipeline stages have already happened. The data has been validated, parsed, encrypted, saved, rendered to PDF, and queued for delivery. But the staff member does not need to know about any of that. They just see "success."

If they want to check whether the email or fax actually got delivered, they can visit the request history page. Each request shows its delivery status: queued, sending, delivered, or failed. These statuses update in the background as the worker processes the jobs.

The takeaway

The system is designed so that the person using it never has to think about queues, encryption, or parsing. They fill in a form, click a button, and the system handles the rest. The complexity is hidden behind a simple interface.
