Module 04 / 07

Security & PHI Protection

How Requests protects patient data at every layer, from login to database to server to audit log. No prior security knowledge needed.

Section 1

Why Security Matters Here

This application handles Protected Health Information (PHI). That means real patient data: names, dates of birth, Medicare numbers, phone numbers, and addresses.

PHI (Protected Health Information)

Any data that can identify a patient or relates to their health. In Requests, this includes the patient's name, DOB, Medicare number, phone number, email, and address.

Australia has specific laws about how this data must be handled. Two are especially important:

  • The Australian Privacy Act and its Australian Privacy Principles (APPs) set the rules for collecting, storing, using, and disclosing personal information. Health information gets extra protections.
  • The Notifiable Data Breaches (NDB) scheme requires organisations to notify the Office of the Australian Information Commissioner (OAIC) and all affected patients if a data breach is likely to cause serious harm.
This is the law

Mishandling patient data is not just a bad look. It carries real legal penalties, mandatory breach notifications, and potential fines. CURA has a legal obligation to protect every piece of PHI this system touches.

Analogy

Think of a hospital vault. You would never leave patient files on a park bench. This app treats digital patient data with the same level of care as a locked medical records room. Every layer of the system is designed to keep that vault sealed.

Section 2

Authentication — Proving Who You Are

Authentication answers one question: are you really who you claim to be? In Requests, this is handled by Auth.js v5 (formerly known as NextAuth).

Auth.js v5

A library for Next.js that handles login sessions, password verification, and multi-factor authentication. It manages the whole "who is this person?" flow so the app does not have to build it from scratch.

Here is what the login process looks like, step by step:

Step 1: Email + Password The user enters their email and password. The password is checked against a bcrypt hash stored in the database (cost factor 12). The actual password is never stored — only its hash.
Step 2: MFA Code (mandatory) After the password is verified, the user must enter a 6-digit code from their TOTP authenticator app (like Google Authenticator). This code changes every 30 seconds.
Step 3: Session Created Only after both checks pass does the user get a session. The session times out after 15 minutes of inactivity.
bcrypt

A one-way hashing algorithm. It turns "MyPassword123" into a scrambled string like "$2b$12$LJ3m..." that cannot be reversed. When you log in, the system hashes what you typed and compares it to the stored hash. Even if someone steals the database, they cannot recover the original passwords.

TOTP (Time-based One-Time Password)

A method for generating short-lived codes. An app on your phone (Google Authenticator, Authy, etc.) uses a shared secret and the current time to generate a 6-digit code. The code is valid for 30 seconds, then a new one appears. The server does the same calculation to verify the code matches.

MFA (Multi-Factor Authentication)

Requiring two or more different types of proof to log in. Factor 1 is "something you know" (your password). Factor 2 is "something you have" (your phone with the authenticator app). Even if someone steals your password, they cannot log in without your phone.

MFA is mandatory for every single user. There are no exceptions. The first time a user logs in, they are forced to set up TOTP by scanning a QR code. They also receive backup codes in case they lose their phone.

Two more protections are built in:

  • Account lockout: 5 failed login attempts locks the account for 30 minutes. This stops attackers from guessing passwords.
  • Session timeout: 15 minutes of inactivity ends your session. You must log in again. This protects against someone walking up to an unattended computer.
Analogy

A two-lock door. Your password is one key. Your phone's authenticator app is the second key. You need both to get in. Even if someone copies one key, the door stays locked without the other.

Section 3

Field-Level Encryption

All patient data in the database is encrypted using pgcrypto, a PostgreSQL extension. But this is not whole-database encryption. It is field-level encryption, which is an important distinction.

pgcrypto

A built-in PostgreSQL extension that provides encryption and hashing functions. Requests uses it to encrypt and decrypt specific columns in the database, right inside PostgreSQL itself.

Field-Level Encryption

Encrypting individual columns (fields) in a database, rather than the entire database. This means a column like "rawPhiInput" is stored as unreadable gibberish. Even someone with direct database access cannot read it without the encryption key.

Two specific fields are encrypted:

  • rawPhiInput — the original patient details text as entered by the user
  • parsedPhi — the structured data extracted from that text (name, DOB, Medicare number, etc.)

Here is the critical part: the encryption key lives outside the database. It is stored in an environment variable on the server. This means:

  • If someone steals a database backup, they get encrypted gibberish. No key, no data.
  • If someone gains access to the server files but not the database, they still cannot read patient data.
  • The key and the data are in different places. An attacker needs both.
Analogy

Each patient file in a lockbox. The filing cabinet (database) might be open, but each individual folder (field) has its own padlock. You need a separate key ring (environment variable) to open any folder. Steal the cabinet, and all you have is a collection of locked boxes.

Why not encrypt everything?

Non-PHI fields like "exam type" or "referring doctor" are not encrypted because the app needs to search and filter by them. Encrypted fields cannot be searched efficiently. The design encrypts what is sensitive and leaves operational data readable for the application to function.

Section 4

Data Residency — Everything Stays in Australia

Under the Australian Privacy Act, there are strict rules about sending personal information overseas. The simplest way to comply is to keep everything in Australia. That is exactly what Requests does.

Component Provider Location
VPS (Server) BinaryLane Sydney, AU
Database PostgreSQL on VPS Sydney, AU
PDF Storage Encrypted on VPS Sydney, AU
Fax Delivery Notifyre (ISO 27001) Australia
Email Delivery Google Workspace SMTP AU domain

Every piece of patient data — from the moment it enters the form to the moment it is archived or deleted — stays within Australian borders. The fax provider, Notifyre, is an Australian company with ISO 27001 certification (an international standard for information security management).

Why this matters

If CURA sent PHI through a US-based fax service, that data would be subject to US laws (like the PATRIOT Act) and could potentially be accessed by foreign governments. Keeping everything in Australia avoids this entirely.

Section 5

Defence in Depth

No single security measure is enough on its own. Requests uses defence in depth — multiple layers of protection stacked on top of each other. If one layer fails, the next one is there to catch the threat.

Defence in Depth

A security strategy that uses multiple independent layers of protection. The idea is that no single layer is trusted to stop every attack. Each layer catches what the one above it might miss.

Here are all the layers, from the outside in:

1
HTTPS + HSTS All traffic encrypted in transit. HSTS tells browsers to never use plain HTTP, even if someone types it.
2
Security Headers CSP (Content Security Policy), X-Frame-Options, X-Content-Type-Options, and Referrer-Policy. These prevent clickjacking, XSS, and data leakage through browser behaviour.
3
Rate Limiting 5 login attempts per IP per 15 minutes. 60 form submissions per hour. Stops brute-force attacks and abuse.
4
CSRF Protection Every form includes a unique token that proves the submission came from the real Requests app, not a malicious website.
5
Input Validation (Zod) Every piece of data hitting the server is validated against a strict schema. Bad data is rejected before it touches the database.
6
Authentication + MFA Password plus TOTP code required for every user, every login. Covered in Section 2.
7
Field-Level Encryption PHI encrypted at rest in the database using pgcrypto. Covered in Section 3.
8
Server Hardening SSH key-only access (no passwords). Fail2ban blocks repeated failed SSH attempts. UFW firewall restricts which ports are open.
9
File Integrity Monitoring Host-level monitoring detects if any system files are unexpectedly changed, which could indicate a compromised server.
CSRF (Cross-Site Request Forgery)

An attack where a malicious website tricks your browser into submitting a form to Requests while you are logged in. CSRF tokens prevent this by requiring a secret value that only the real Requests page knows.

Zod

A TypeScript library for defining exactly what shape data should be (e.g., "this field must be a string between 1 and 100 characters"). If data does not match the schema, it is rejected. This runs on the server, not just in the browser.

Fail2ban

A tool that watches server logs for repeated failed login attempts (like SSH brute-force attacks). After too many failures from the same IP address, it automatically blocks that IP.

UFW (Uncomplicated Firewall)

A firewall that controls which network ports are open on the server. Only the ports that Requests actually needs (HTTP, HTTPS, SSH) are open. Everything else is blocked.

Section 6

Audit Logging — The Paper Trail

Every action in Requests is recorded. Not vaguely — specifically. This is the audit trail, and it exists for two reasons: investigating incidents and proving compliance.

Requests uses two-tier logging:

  • Audit log (database) — Who did what, to which resource, when, and from which IP address. Stored in a database table.
  • Application log (file) — Technical events like errors, job processing results, and system health. Stored in log files on the server.
Critical rule

Audit logs NEVER contain PHI. Not patient names. Not DOBs. Not Medicare numbers. Only record IDs. If an audit log entry says "User viewed request #4521", the log does not say whose request that was. You would need database access (and the encryption key) to look that up.

What the audit log records for every action:

  • Who — the user ID (or API key ID for programmatic access)
  • What — the action performed (created, viewed, updated, deleted, exported)
  • Which — the resource ID (request #4521, not "John Smith's CT scan")
  • When — timestamp
  • From where — IP address
Analogy

Security cameras in the vault. They record who entered, when, and which drawer they opened — but they cannot see what is inside the folders. The cameras (audit logs) know "Dr. Harrison accessed drawer #4521 at 3:14 PM from the main office." They do not know that drawer #4521 contains Mrs. Smith's MRI request.

Two more important details:

  • Audit logs are never purged. Even after patient data is deleted, the record of who accessed it remains forever.
  • Off-box log shipping. Logs are copied to a separate server in real time. This means even if an attacker compromises the main server and tries to delete their tracks, the logs survive on the other machine.
Off-box Log Shipping

Sending copies of log files to a different server as they are written. This provides tamper resistance — an attacker who gains access to the main server cannot delete logs that have already been shipped elsewhere.

Section 7

Data Retention

Patient data does not live in the system forever. Requests follows a two-tier retention policy that balances operational needs with privacy obligations.

0 – 90 days: Active Patient data lives in the PostgreSQL database, encrypted at field level. Accessible to authorised users through the app for follow-ups, resending, and auditing.
90 days – 7 years: Archived Data is moved out of the active database and archived using GPG encryption. It is no longer accessible through the app. It exists only for compliance purposes — in case a regulatory body needs historical records.
After 7 years: Securely deleted Archived data is permanently destroyed. The 7-year retention period aligns with Australian medical records requirements.
GPG Encryption

GNU Privacy Guard — a widely-used encryption tool. When data is archived, it is encrypted with GPG before being stored. This is a separate layer of encryption on top of the field-level encryption already applied in the database.

There is one exception to the retention schedule:

Audit logs are forever

Audit logs are never deleted. Even after the patient data they reference has been archived and eventually destroyed, the record of who accessed what and when remains permanently. This is essential for compliance and incident investigation.
