Module 02 · ~12 min

The Data Model

Where does all the information live? In this module, you will learn about the database tables that store everything from patient details to delivery tracking.

1. Why a Database?

When a doctor submits an imaging request, the system needs to remember it. Who was the patient? Which radiology practice? Did the fax go through? All of this has to live somewhere permanent, not just in the browser.

That somewhere is a database — a structured way to store and retrieve data.

Analogy
Think of a filing cabinet. Each drawer is a table. Each folder in the drawer is a row (one record). The label on each folder tells you what is inside — that is a column (a field). When you want to find a patient's imaging request, you open the right drawer, look at the labels, and pull the folder.

Requests uses two key technologies for its database layer:

PostgreSQL — The Database
What is it? An open-source database engine. Free, battle-tested, used by major companies worldwide.
Why this one? Supports pgcrypto — a built-in extension for encrypting individual fields. Critical for patient data.
Prisma — The ORM
What is it? A tool that lets TypeScript code talk to the database without writing raw SQL.
What is an ORM? Object-Relational Mapper. Translates between code objects and database rows.
Why it helps Developers write TypeScript instead of SQL. Prisma generates the SQL behind the scenes.
Jargon buster
SQL (Structured Query Language) is the language databases understand. Instead of writing SQL directly, Prisma lets us write TypeScript and it translates. Think of Prisma as an interpreter between two people who speak different languages.

2. The Eight Tables

The Requests database has eight tables. Each one stores a different type of information. Here is a bird's-eye view:

User
Staff accounts and login details
Provider
Referring doctors + provider numbers
RadiologyPractice
Where requests get sent
ImagingRequest
The core table — one row per request
DeliveryJob
Tracks each delivery attempt
AuditLog
Who did what, when
Session
Login sessions
ApiKey
Keys for Synaptum 8 integration

Let us look at each table in detail.

User

Stores staff accounts. There are 3 doctors and 1 admin. Each user must have MFA (multi-factor authentication) enabled to log in.

Jargon buster
MFA / TOTP — Multi-Factor Authentication means you need something beyond your password to log in (like a code from an authenticator app on your phone). TOTP stands for Time-based One-Time Password — the 6-digit code that changes every 30 seconds.

This is the heart of the system. Every time a doctor submits a request form, one row is created here. Notice the two encrypted fields — those hold patient health information.

Why two PHI fields?
rawPhiInput is exactly what the doctor typed into the patient details box. parsedPhi is the structured version after the system has extracted the name, date of birth, Medicare number, phone, etc. Both are stored encrypted because they contain patient health information.

DeliveryJob

One imaging request can produce multiple delivery jobs. For example: an email to the radiology practice, a fax to the radiology practice, a filing copy to CURA, and optionally an email to the patient. Each one is tracked separately.

Every action in the system is recorded here. Who logged in? Who viewed a request? Who changed a radiology practice? It records the action and a reference ID, but never any patient data.

Tracks who is currently logged in. When you log in, a session row is created. When it expires or you log out, it is removed.

ApiKey

Synaptum 8 (the future practice management system) can create imaging requests through an API instead of the web form. Each API key is stored as a hash, not the actual key, so even a database breach would not reveal the real key.

Jargon buster
SHA-256 hash — A one-way mathematical function that converts data into a fixed-length string of characters. You can turn a key into a hash, but you cannot turn a hash back into the key. This means the real API key is shown once when created, then never stored again.

3. How the Tables Connect

Tables do not exist in isolation. They are linked together using relationships. A relationship means one table references another using an ID field.

Analogy
Imagine a folder in the "Imaging Requests" drawer has a sticky note that says "See Provider #4" and another that says "See User #2". You would go to the Provider drawer and pull folder #4, then the User drawer and pull folder #2. That is a relationship — one record points to another.

Here are the connections between tables:

Relationships
User → creates many → ImagingRequest
User → has default → Provider
Provider → used in many → ImagingRequest
RadiologyPractice → receives many → ImagingRequest
ImagingRequest → has many → DeliveryJob
User → generates many → AuditLog
User → owns many → ApiKey

The key pattern here is "one-to-many". One user creates many requests. One request creates many delivery jobs. One radiology practice receives many requests. The "many" side stores the ID of the "one" side.

Jargon buster
One-to-many — A relationship where one record in table A can be linked to many records in table B. Example: one User can create 100 ImagingRequests, but each ImagingRequest has exactly one creator.

4. The Encryption Story

Two fields in the ImagingRequest table — rawPhiInput and parsedPhi — contain patient health information. Names, dates of birth, Medicare numbers, phone numbers. This data is encrypted using a PostgreSQL extension called pgcrypto.

Analogy
Think of a lockbox inside the filing cabinet. Someone might break into the office and open the filing cabinet. But each patient folder is inside its own padlocked box. Without the key to those lockboxes, the folders are unreadable.

What does "field-level encryption" mean? It means individual columns (fields) are encrypted, not just the entire database. Most of the data in the database — like exam types, delivery statuses, and provider numbers — is stored normally. Only the fields that contain patient information get the extra layer of encryption.

Where is the encryption key? The key used to encrypt and decrypt patient data is stored separately from the database itself. It lives in the server's environment variables. This means:

  • If someone steals the database file, they cannot read patient data.
  • If someone gets the encryption key but not the database, they have nothing to decrypt.
  • You need both to access PHI.
API keys are hashed too
The ApiKey table stores keyHash instead of the actual key. This uses SHA-256, a one-way hash. Even if the database is stolen, the attacker cannot recover the real API keys. The actual key is shown once when created, then never stored again.

5. Status Tracking

Each ImagingRequest has a status field. But this status is not set manually — it is derived (calculated) from the statuses of its delivery jobs. Here is how:

Request Status What It Means Delivery Job States
pending Still in progress Jobs are queued or processing
delivered Everything succeeded All jobs completed successfully
partial Some worked, some did not Some jobs succeeded, some failed
failed Nothing got through All jobs failed

For example: a request sent via both email and fax. The email goes through, but the fax fails. The request status becomes partial because some delivery jobs succeeded and some did not.

Why this matters
The dashboard shows doctors the status of their requests at a glance. A partial status tells them something needs attention — maybe the fax number was wrong, or the radiology practice's fax machine was offline.
